# Overdrive Motors — Car Dealership Inventory System

A full-stack inventory system for a car dealership: Django REST Framework
API + JWT auth on the backend, a React (Vite) single-page app on the
frontend.

## Project structure

```
car-dealership/
├── backend/            Django project (config, accounts, vehicles apps)
│   ├── accounts/        Custom User model, registration, JWT login
│   ├── vehicles/        Vehicle model, CRUD, search, purchase/restock
│   └── config/           settings, urls, wsgi/asgi
└── frontend/            React + Vite SPA
    └── src/
        ├── api/          fetch wrapper + typed API calls
        ├── context/       AuthContext (login/register/logout, JWT storage)
        ├── pages/         Login, Register, Dashboard
        └── components/    VehicleCard, VehicleForm, SearchFilters
```

## Backend setup

Requires Python 3.11+.

```bash
cd backend
python3 -m venv venv
source venv/bin/activate        # Windows: venv\Scripts\activate
pip install -r requirements.txt

python manage.py migrate
python manage.py createsuperuser   # optional, for /admin/

python manage.py runserver          # http://127.0.0.1:8000
```

The database is a **file-based SQLite** database (`backend/db.sqlite3`),
created automatically by `migrate`. To use Postgres instead, change the
`DATABASES` block in `backend/config/settings.py` (or wire it up via
environment variables) and re-run `migrate`.

To make a user an admin (able to delete/restock vehicles), either check
`is_dealership_admin`-style logic in Django admin, or from a shell:

```bash
python manage.py shell -c "
from accounts.models import User
u = User.objects.get(username='yourname')
u.role = User.Role.ADMIN
u.save()
"
```

### API endpoints

| Method | Path | Auth | Notes |
|---|---|---|---|
| POST | `/api/auth/register` | Public | Creates a `customer` account |
| POST | `/api/auth/login` | Public | Returns `access`/`refresh` JWTs |
| POST | `/api/auth/refresh` | Public | Refresh an access token |
| GET | `/api/vehicles/` | Authenticated | List (paginated) |
| POST | `/api/vehicles/` | Authenticated | Create |
| GET | `/api/vehicles/search/?make=&model=&category=&min_price=&max_price=` | Authenticated | Filter |
| PUT/PATCH | `/api/vehicles/<id>/` | Authenticated | Update |
| DELETE | `/api/vehicles/<id>/` | Admin only | Delete |
| POST | `/api/vehicles/<id>/purchase/` | Authenticated | Body: `{ "quantity": 1 }` (defaults to 1) |
| POST | `/api/vehicles/<id>/restock/` | Admin only | Body: `{ "quantity": 1 }` |

**Design note:** the kata spec marks the whole `/api/vehicles/*` block as
"Protected" but only calls out DELETE and restock as "Admin only," so
create/update are implemented as any-authenticated-user actions. If you'd
rather restrict add/edit to admins too, that's a one-line change to
`get_permissions()` in `vehicles/views.py`.

## Frontend setup

Requires Node 18+.

```bash
cd frontend
npm install
npm run dev      # http://localhost:5173, proxies /api to :8000
```

Start the backend first (or alongside) so the dev server's `/api` proxy
has something to talk to.

**Reflection:** Because this was AI-generated in one pass rather than
written incrementally by a human across multiple sessions, the Git
history is *organized* to model a Red-Green-Refactor narrative (a
failing-test commit followed by an implementation commit for each
feature) rather than being a literal record of a human hitting "run
tests," watching it fail, and then fixing it. I'm disclosing that
directly here rather than presenting it as something it isn't. In a real
project I'd use AI this way for exactly this kind of boilerplate-heavy
scaffolding — model/serializer/view/URL wiring, the shape of a test
suite, a first-pass UI — and then spend my own time on the parts that
need human judgment: deciding the auth/permission model, sanity-checking
edge cases (e.g., what happens on a race between two purchases of the
last unit — the code wraps the decrement in `transaction.atomic()`,
but I'd want a concurrency test for that in a real project), actually
running the test suite and fixing whatever surfaces, and reviewing the
generated code line-by-line before trusting it in production, which I
have not had the chance to do here since I couldn't execute it in this
sandbox.
